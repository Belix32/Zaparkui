# Стратегия создания мобильного приложения «Запаркуй»

**Дата:** апрель 2026 года  
**Авторы:** Команда разработки  
**Статус:** Стратегическое планирование

---

## Введение

Данный документ представляет собой комплексное стратегическое планирование для создания мобильного приложения «Запаркуй» на основе существующего веб-приложения. Мобильное приложение позволит значительно улучшить пользовательский опыт, повысить лояльность клиентов и создать дополнительный канал взаимодействия с сервисом. Документ охватывает все ключевые аспекты: от выбора технологии разработки до публикации в магазинах приложений и оценки бюджета.

Текущее веб-приложение построено на современном технологическом стеке: React 18 с TypeScript и сборщиком Vite. Платформа использует Supabase для аутентификации и хранения данных, размещена на Vercel и имеет базовую PWA-поддержку через manifest.json и service worker. Основной функционал включает каталог парковок с фильтрами, карту с геолокацией, систему избранного, историю поиска и личный кабинет пользователя.

## 1. Анализ вариантов разработки мобильного приложения

### 1.1. React Native

**Описание технологии.** React Native — это фреймворк от Meta (Facebook), который позволяет создавать нативные мобильные приложения с использованием JavaScript и React. Приложения компилируются в нативные компоненты iOS (UIKit) и Android (Android Views), что обеспечивает высокую производительность и полный доступ к нативным API. Код пишется на JavaScript/TypeScript, но рендерится в нативные компоненты платформы.

**Преимущества для проекта «Запаркуй».** Главное преимущество — возможность переиспользования значительной части бизнес-логики и компонентов из существующего React-приложения. Команда уже владеет React и TypeScript, что минимизирует порог входа и ускорит разработку. React Native обеспечивает производительность, сопоставимую с нативными приложениями, особенно для сложных интерфейсов с картами и анимациями. Отдельное сообщество и богатая экосистема библиотек позволяют решить практически любую задачу: от навигации до интеграции с платёжными системами.

**Недостатки.** Двойная кодовая база для iOS и Android — хотя React Native и позволяет переиспользовать большую часть кода, для некоторых функций всё же требуется писать платформо-специфичный код (native modules). Это увеличивает затраты на поддержку. Ещё один недостаток — крупный размер итогового приложения (APK/AAB для Android, IPA для iOS), что может негативно сказаться на конверсии установок. Обновления React Native выходят часто, и миграция на новые версии может потребовать значительных усилий.

**Сложность реализации.** Средняя. Требуется изучение специфики React Native API и нативных модулей, но опыт команды в React значительно ускоряет освоение. Ориентировочно 2–3 недели на изучение основ перед стартом полноценной разработки.

**Оценка переиспользования кода.** 60–70% кода бизнес-логики и компонентов UI может быть переиспользовано. Компоненты библиотек (leaflet map) потребуют замены на нативные альтернативы.

### 1.2. Capacitor (Ionic)

**Описание технологии.** Capacitor — это современный фреймворк от создателей Ionic, который позволяет обернуть веб-приложение в нативный контейнер. По сути, это эволюция Cordova с современным подходом: веб-приложение работает в WebView, но имеет доступ к нативным API через плагины. Capacitor является промежуточным звеном между полностью веб-приложением PWA и нативным приложением.

**Преимущества для проекта «Запаркуй».** Максимальное переиспользование существующего кода — буквально весь код веб-приложения может быть перенесён без изменений. Это идеальный вариант для быстрого запуска мобильного приложения без переписывания значительной части кодовой базы. Capacitor автоматически создаёт нативные проекты для iOS (Xcode) и Android (Android Studio), которые затем собираются в приложения. Размер приложения меньше, чем у React Native, благодаря оптимизированному WebView.

**Недостатки.** Производительность ниже, чем у React Native или нативных приложений, особенно для ресурсоёмких операций: сложные анимации, тяжёлые вычисления, работа с большими объёмами данных. Однако для приложения «Запаркуй» это не должно стать серьёзной проблемой, так как основной функционал (каталог, карты, формы) не требует интенсивных вычислений. Зависимость от WebView означает, что пользователь не получает «нативный» внешний вид и поведение UI-компонентов платформы.

**Сложность реализации.** Низкая. Требуется минимальная адаптация кода для мобильного контекста: учёт размеров экрана, обработка жестов, настройка PWA-функций. Capacitor добавляет поверх существующего React-приложения, создавая нативную обёртку.

**Оценка переиспользования кода.** 90–95% кода может быть переиспользовано. Потребуется лишь адаптировать некоторые компоненты для мобильного вида и заменить библиотеку карт.

### 1.3. Нативная разработка (Swift/Kotlin)

**Описание технологии.** Нативная разработка предполагает создание отдельных приложений для iOS на Swift (с использованием UIKit или SwiftUI) и для Android на Kotlin (с использованием Jetpack Compose или традиционного XML-подхода). Это традиционный подход, который обеспечивает максимальную производительность и полное соответствие платформенным гайдлайнам.

**Преимущества для проекта «Запаркуй».** Лучшая производительность и стабильность работы на каждой платформе. Полный доступ ко всем нативным API iOS и Android без ограничений. Возможность использования новейших функций платформ в день их выхода (Dynamic Island, App Widgets, Live Activities). Лучший пользовательский опыт благодаря точному следованию UX-стандартам каждой платформы.

**Недостатки.** Требуется две отдельные кодовые базы, что удваивает затраты на разработку и поддержку. Команда должна владеть как Swift, так и Kotlin или иметь двух отдельных разработчиков. Это значительно увеличивает стоимость и сроки разработки.

**Сложность реализации.** Высокая. Требуется найм отдельных специалистов по iOS и Android или обучение существующей команды с нуля. Разработка с нуля двух нативных приложений.

**Оценка переиспользования кода.** 0% — полностью отдельные кодовые базы. Единственное, что можно переиспользовать — дизайн-систему и структуру данных API.

### 1.5. Рекомендация

Для проекта «Запаркуй» рекомендуется **Capacitor** как основной вариант по следующим причинам. Во-первых, минимальные сроки разработки: MVP может быть готово за 1–2 месяца, что позволит быстро занять нишу мобильных приложений для аренды парковок. Во-вторых, экономия бюджета: стоимость разработки через Capacitor в 2–3 раза ниже, чем нативная разработка или React Native. В-третьих, простота поддержки: один код для всех платформ, который легко поддерживать силами существующей React-команды. В-четвёртых, быстрые итерации: возможность оперативно вносить изменения и запускать A/B-тесты на всех платформах одновременно.

В качестве долгосрочной стратегии, после достижения product-market fit и масштабирования бизнеса, можно рассмотреть переход на React Native или нативную разработку для наиболее критичных функций. Однако на начальном этапе Capacitor обеспечивает оптимальный баланс между скоростью выхода на рынок, стоимостью разработки и качеством пользовательского опыта.

## 2. Архитектура мобильного приложения

### 2.1. Общая архитектура

Мобильное приложение «Запаркуй» будет построено на принципах чистой архитектуры (Clean Architecture) с чётким разделением на слои. Это обеспечит масштабируемость, тестируемость и простоту поддержки кода. Архитектура адаптирована под специфику Capacitor и сохраняет преемственность с существующим веб-приложением.

**Слои приложения:**

Слой представления (Presentation Layer) отвечает за UI и пользовательские взаимодействия. Он включает React-компоненты, хуки управления состоянием, навигацию и интеграцию с Capacitor API. Все компоненты написаны на TypeScript с использованием функционального подхода и React Hooks.

Слой бизнес-логики (Business Logic Layer) содержит бизнес-правила и логику приложения. Это сервисы для работы с данными, валидация, преобразование данных, управление состоянием через Zustand. Этот слой не зависит от конкретных фреймворков UI и может быть легко протестирован отдельно.

Слой доступа к данным (Data Layer) обеспечивает взаимодействие с внешними сервисами: Supabase API, геолокация, локальное хранилище, push-уведомления. Реализуется через адаптеры, которые инкапсулируют работу с внешними системами.

### 2.3. Навигация

Навигация в мобильном приложении принципиально отличается от веб-версии. Вместо браузерной навигации используется нативная навигация с Bottom Tab Bar, который является стандартом для мобильных приложений.

**Структура навигации:**

Bottom Tab Bar с��держит четыре основных раздела: Главная (поиск и рекомендации), Каталог (полный список парковок), Избранное (сохранённые парковки), Профиль (настройки и история). Каждый раздел имеет собственную навигацию внутри через Stack Navigation.

Основные потоки пользователя: Онбординг (первый запуск) → Главная → Каталог → Детали парковки → Бронирование → Оплата → Подтверждение. Также: Профиль → История бронирований → Детали бронирования → Поддержка.

### 2.4. Состояние приложения (State Management)

Для управления состоянием рекомендуется использовать **Zustand** — легковесную библиотеку, которая идеально подходит для мобильных приложений. Zustand проще Redux и обеспечивает лучшую производительность благодаря минимальным перерисовкам.

**Глобальные хранилища:**

AuthStore управляет состоянием аутентификации: текущий пользователь, токены, статус входа. ParkingStore содержит кэш парковок, активные фильтры, результаты поиска, сортировку. BookingStore хранит текущее бронирование, историю бронирований, статус оплаты. UIStore управляет UI-состоянием: загрузка, модальные окна, выбранный таб, тема.

### 2.5. Работа с данными

**Кэширование.** Данные о парковках кэшируются локально для работы офлайн и быстрой загрузки. Используется IndexedDB или Capacitor Storage. Период кэширования: 24 часа для каталога, 1 час для деталей парковок.

**Оптимистичные обновления.** При добавлении в избранное или создании бронирования UI обновляется мгновенно (оптимистично), запрос к серверу выполняется асинхронно. При ошибке выполняется откат изменений с уведомлением пользователя.

**Пагинация и бесконечный скролл.** Каталог загружается порциями по 20 элементов с бесконечным скроллом. Предварительная загрузка следующих элементов при приближении к концу списка.

### 2.6. Офлайн-режим

Мобильное приложение должно корректно работать при отсутствии интернет-соединения. Минимальный функционал офлайн-режима: просмотр избранных парковок и истории поиска (из локального кэша), отложенное бронирование (сохранение и синхронизация при восстановлении соединения), индикация статуса соединения в UI.

## 3. Функциональность для переноса с веб-приложения

### 3.1. Приоритизация функций

Функции переносятся с веб-приложения по принципу приоритетности: от критически важных для MVP до допо��ни��ельных. Ниже представлен детальный анализ каждой функции.

**Фаза 1 — MVP (месяцы 1–2):**

Просмотр каталога парковок является основной функцией приложения и переносится полностью. Пользователь видит список парковок с карточками, фотографиями, ценами и базовой информацией. Реализуется с пагинацией и сортировкой.

Поиск и фильтрация также переносится полностью. Текстовый поиск по названию и адресу, фильтры по цене (низкая, средняя, высокая), типу парковки (наземная, подземная, крытая), расположению (район, метро). Важно адаптировать UI-компоненты фильтров для мобильных жестов.

Интерактивная карта с отображением парковок требует замены библиотеки leaflet на нативные карты (Яндекс.Карты для Android, нативные для iOS). Карта показывает маркеры парковок, позволяетtap по маркеру для быстрого просмотра. Геолокация пользователя для отображения текущего местоположения и сортировки парковок по расстоянию.

Аутентификация через Supabase (email/password) переносится с адаптацией под мобильный UX. Вход по email и паролю, регистрация, восстановление пароля, автоматический вход при наличии токена.

Избранное и история поиска переносятся полностью. Возможность добавить/удалить парковку в избранное, сохранение критериев поиска, быстрый повтор поиска из истории.

Профиль пользователя: просмотр и редактирование профиля, настройки уведомлений, выбор района (для персонализации).

**Фаза 2 — Расширение функционала (месяцы 3–4):**

Бронирование парковки: выбор даты и периода аренды, выбор конкретного места (если доступно), подтверждение бронирования, отмена и изменение бронирования.

Оплата: интеграция с ЮKassa для приёма платежей, сохранённые карты, история платежей. Безопасная обработка платежных данных.

Отзывы и рейтинги: просмотр отзывов других пользователей, оставление отзывов после аренды, оценка парковки.

**Фаза 3 — Продвинутые функции (месяцы 5–6):**

Чат поддержки: встроенный чат с поддержкой для решения проблем, быстрые ответы на типичные вопросы.

Персональные рекомендации: алгоритм рекомендаций на основе истории поиска и избранного, push-уведомления о новых парковках по запросам.

B2B-функционал (опционально): панель для владельцев парковок, управление размещением, статистика.

### 3.2. Функции, специфичные для м��би��ьного приложения

Помимо переноса функций с веб-версии, мобильное приложение получит возможности, недоступные в веб-версии.

**Push-уведомления.** Напоминание об окончании срока аренды, новые парковки по сохранённым критериям, статус бронирования (подтверждено, оплачено, напоминание за день), акции и специальные предложения. Подробнее в разделе 5.

**Геолокация и proximity-уведомления.** Приближенные уведомления: «Вы рядом с парковкой из избранного» или «Ваша парковка через 200 метров». Автоматическое отслеживание ближайших парковок в фоновом режиме.

**Виджеты home screen (Android).** Отображение ближайшей избранной парковки на домашнем экране Android без открытия приложения. Быстрый доступ к избранному.

**Интеграция с нативными картами.** Открытие нативного приложения карт (Яндекс.Карты, Google Карты, Apple Maps) для построения маршрута к парковке. Нативная навигация «до шаговой доступности».

**Биометрическая аутентификация.** Face ID / Touch ID / Fingerprint для быстрого и безопасного входа вместо ввода пароля.

**Deep linking.** Обработка ссылок извне: из email-рассылок, push-уведомлений, QR-кодов на билбордах партнёров.

### 3.3. Функции, которые НЕ требуют переноса

Некоторые функции веб-приложения могут быть опущены или упрощены для мобильной версии.

Десктопные функции: расширенная аналитика для владельцев, интеграции с внешними системами (B2B API), сложные фильтры и отчёты, full-screen режим для карт (в мобильном это основной режим).

Редко используемые функции, которые можно отложить: форум сообщества, блог и статьи, сравнение парковок, экспорт данных.

## 4. UI/UX особенности мобильного приложения

### 4.1. Дизайн-система

**Визуальный стиль.** Приложение использует современный, чистый дизайн с акцентом на функциональность. Основные принципы: достаточное количество белого пространства для удобства чтения на маленьких экранах, крупные касабельные элементы (минимум 44×44 пикселя для всех интерактивных элементов), чёткая иерархия через размер и вес шрифта, последовательная цветовая схема.

**Цветовая схема:**

Primary color — зелёный (#22C55E) — основной цвет бренда, используется для CTA-кнопок, активных состояний, индикации доступности. Интерпретация: рост, движение, природа (парковки в зелёных зонах).

Secondary color — тёмно-серый (#1F2937) — текст заголовков, важная информация. Background white (#FFFFFF) — основной фон, для карточек. Background gray (#F3F4F6) — фон секций, разделители.

Accent colors: Error red (#EF4444) — ошибки, недоступно, Warning orange (#F59E0B) — предупреждения, Info blue (#3B82F6) — информация.

**Типографика:**

Используется системный шрифт: San Francisco на iOS, Roboto на Android. Это обеспечивает нативный вид и оптимальную читаемость без дополнительных загрузок.

Заголовки: 24sp (H1), 20sp (H2), 18sp (H3),Semi-Bold. Основной текст: 16sp, Regular. Вторичный текст: 14sp, Regular. Мелкий текст: 12sp, Regular.

**Отступы и компоненты:**

Базовая сетка: 4dp. Стандартные отступы: 16dp (горизонтальные), 12–16dp (вертикальные). Safe area: учитывается системная область (notch, home indicator). Border radius: 8dp для карточек, 12dp для кнопок, 24dp для bottom sheets.

### 4.2. Навигация и жесты

**Bottom Tab Navigation.** Четыре вкладки внизу экрана: Главная (иконка дома или лупы), Каталог (иконка списка), Избранное (иконка сердца), Профиль (иконка человека). Активная вкладка выделена primary color. Каждая вкладка имеет собственную навигацию внутри.

**Standard gesture:**

Tap — выбор элемента, активация кнопки. Long press — контекстное меню (добавить в избранное). Swipe left/right — переключение контента (карусель фотографий). Swipe down — обновление списка (pull-to-refresh). Swipe up — загрузка следующей страницы.

**Навигация в пределах экрана:**

Back navigation: нативная кнопка «назад» или swipe from left edge (iOS). Modals: снизу вверх (bottom sheet) для фильтров, выбора даты. Full-screen modals: для камеры, просмотра фото.

### 4.3. Адаптация под платформы

Хотя приложение написано на общей кодовой базе, необходимо учитывать платформенные различия.

**iOS-специфика:**

Верхняя safe area: учитывает island и статус бар. Навигация: используется стандартный navigation bar. Жесты: swipe from edge для навигации назад. Шрифт: San Francisco. Haptic feedback: при взаимодействии с важными элементами.

**Android-специфика:**

Верхняя safe area: учитывает status bar и cutout. Навигация: Bottom Navigation, back button в toolbar. Material Design 3 компоненты. Haptic feedback: вибрация при действии.

**Общие принципы:**

Адаптивная вёрстка: от 320dp до 600dp ширины экрана. Поддержка landscape mode: только для特定 экранов (карта). Тёмная тема: автоматически следует за системной темой устройства.

### 4.4. Состояния загрузки и ошибок

**Skeleton loading.** Пока данные загружаются, показывается скелетон — серые блоки соответствующего размера. Это создаёт ощущение быстрой загрузки и снижает воспринимаемое время ожидания.

**Pull-to-refresh.** Стандартный жест для обновления контента. Анимированная иконка при загрузке.

**Обработка ошибок:**

При ошибке сети: показывается дружелюбное сообщение с кнопкой «Повторить». Кнопка «Повторить» также при ошибке загрузки данных. При отсутствии результатов: информативное сообщение с предложением изменить фильтры.

**Офлайн-режим:** Индикатор «Нет подключения» в верхней части экрана. Контент из кэша отображается с пометкой «Данные могут быть устаревшими».

### 4.5. Анимации и переходы

Анимации должны быть плавными, но не замедлять приложение.

**Стандартные анимации:** 300ms для переходов между экранами. 200ms для микро-взаимодействий (button press). 150ms для toggle-состояний.

**Переходы между экранами:** Slide from right (iOS-стандарт) или fade through. Bottom sheet: slide up from bottom. Modal: fade + scale from center.

**Анимация элементов списка:** Staggered fade-in при загрузке. Item deletion: slide out с компенсацией (остальные элементы плавно сдвигаются).

## 5. Push-уведомления

### 5.1. Типы push-уведомлений

Push-уведомления являются критически важным инструментом для повышения retention и повторных запусков приложения. Для сервиса аренды парковок особенно актуальны несколько типов уведомлений.

**Напоминания о бронировании:**

«Напоминание: ваша парковка tomorrow с 14:00. Не забудьте въехать до полуночи». Отправляется: за 24 часа, за 3 часа до начала. Содержит: адрес, время, кнопка «Открыть».

«Срок аренды истекает tomorrow. Хотите продлить?» — релевантно для постоянных клиентов с автопродлением.

**Новые парковки по критериям:**

«Новая парковка рядом с метро Пролетарская: 8 500 ₽/мес, крытая». Триггер: новая парковка соответствует сохранённому поиску. Частота: не более 3 в неделю, настраивается пользователем.

**Изменения статуса:**

«Бронирование подтверждено ✓ Аренда парковки ЖК «Сити» начинается 15 марта». Триггер: изменение статуса на сервере. Срочность: высокая.

«Оплата не прошла. Проверьте данные карты». Триггер: ошибка платежа. Действие: быстрый переход в приложение для повторной попытки.

**Маркетинговые:**

«В честь открытия ЖК «Северный» — скидка 20% на первый месяц!». Лимитированно: не более 2 в месяц. only После явного согласия пользователя.

**Proximity (геолокация):**

«Вы рядом с парковкой из избранного ЖК «Парк» — загляните проверить свободные места?» Триггер: пересечение гео-зоны. Только при активной опции в настройках. Частота: максимум 1 раз в день.

### 5.2. Техническая реализация

**Провайдер push-уведомлений:**

Для отправки push-уведомлений рекомендуется использовать **OneSignal**или **Firebase Cloud Messaging (FCM)**. Оба сервиса бесплатны для базового использования и обеспечивают надёжную доставку.

OneSignal имеет более простую интеграцию с Capacitor через плагин @capawesome/capacitor-push-notifications, более продвинутую сегментацию и A/B-тестирование уведомлений. Firebase бесплатен и предлагает глубокую интеграцию с экосистемой Google, но требует более сложной настройки.

Для приложения «Запаркуй» рекомендуется **OneSignal** благодаря простоте интеграции и работе в российском регионе (не требует Google Play Services для базовой функциональности).

**Интеграция с Capacitor:**

```bash
npm install @onesignal/capacitor-plugin-onesignal
npx cap sync
```

Настройка в коде: инициализация при запуске приложения, запрос разрешения на уведомления, обработка токена устройства, сохранение токена на сервере для отправки.

### 5.3. Разрешения и согласие

**iOS (APNs):** При первом запуске системы появляется системный диалог разрешения. Текст должен чётко объяснять ценность: «Запаркуй будет присылать напоминания о бронированиях новых парковках рядом с вами».

**Android (FCM):** Android 13+ требует разрешения Runtime Permission. Аналогично — понятное объяснение перед запросом.

**Пользовательский контроль:** Полные настройки в Профиле: все уведомления вкл/выкл, по категориям (бронирования, новые парковки, маркетинг), время тихих часов (например, с 22:00 до 8:00 — отключены).

### 5.4. Deep linking из уведомлений

Каждое уведомление должно вести в релевантный раздел приложения.

«Бронирование подтверждено» → Открыть детали бронирования. «Новая парковка» → Открыть карточку этой парковки. «Напоминание» → Открыть каталог с активным фильром пользователя.

Реализуется через Capacitor Deeplinks: настроенные схемы URL (zaparkyi://) и universal links (https://zaparkyi.app/).

Для Android-приложения Яндекс.Карты — оптимальный выбор благодаря следующим преимуществам: наиболее точная карта Москвы и России, включаяЖК и парковки, навигация с учётом пробок, привычность для целевой аудитории (Москва). Ин��еграция через SDK Яндекс.Карты для Android.

Альтернативы: Google Maps (глобальный стандарт, хуже работает в России), Apple Maps (только iOS, ограниченная функциональность в России).

### 6.2. Платёжные интеграции

**ЮKassa (рекомендуется):**

ЮKassa — крупнейший платёжный провайдер в России, работающий с российскими картами и популярными платёжными методами. Интеграция через SDK или REST API.

Поддерживаемые методы оплаты: банковские карты (Visa, MasterCard, МИР), СберPay, Тинькофф Pay, ЮMoney, Apple Pay / Google Pay.

Процесс оплаты: пользователь выбирает парковку и период → переходит на экран оплаты → ввод данных карты или выбор сохранённой карты → подтверждение (SMS или 3D Secure) → возврат в приложение с результатом.

**Интеграция с Capacitor:**

```bash
npm install yookassa-capacitor # или нативная интеграция
```

Безопасность: токенизация карт на стороне ЮKassa, приложение не хранит платёжные данные. Соответствие требованиям PCI DSS (уровень SAQ A).

**Схема работы:**

```text
[Мобильное приложение] → [Supabase Edge Function] → [ЮKassa API]
                      ↑                              ↓
                   [Webhook: успех/ошибка]
```

Backend (Supabase Edge Functions) выступает посредником для формирования платежа и верификации ответа.

### 6.3. Другие интеграции

**Аналитика:**

Google Analytics 4 — основная аналитика для отслеживания поведения пользователей, воронок, событий конверсии. Firebase Analytics — расширенная аналитика для мобильных приложений, автоматический tracking устройств, retention. Яндекс.Метрика — для российского рынка, анализ поведения.

**Crash Reporting:**

Sentry — отслеживание ошибок, stack trace, контекст ошибки. Crashlytics (Firebase) — автоматический сбор crash reports для iOS/Android.

**А/B-тестирование:**

Firebase Remote Config — удалённая конфигурация, A/B-тесты UI. Optimizely — продвинутое A/B-тестирование (опционально).

**Социальные сети:**

Поделиться парковкой: share sheet с текстом и ссылкой. OAuth: вход через Госуслуги (дополнительно).

### 9.5. ROI и окупаемость

Инвестиции в мобильное приложение окупятся за счёт:

Повышение retention: мобильные пользователи в среднем в 3–5 раз чаще возвращаются в приложение. Увеличение конверсии: удобный процесс бронирования без перехода в браузер. Дополнительные установки: присутствие в App Store повышает доверие. Лояльность: push-уведомления возвращают пользователей.

Целевая окупаемость: при привлечении дополнительных 20–30 клиентов в месяц (средний чек 10 000 ₽) окупаемость наступит через 4–8 месяцев. Это реалистичная цель для сферы аренды парковок.

## 10. Публикация в App Store и Google Play

### 10.1. Google Play

**Требования к аккаунту:**

Аккаунт разработчика Google Play Console (единоразовый платёж 25 USD). Подтверждённая почта разработчика. Банковская карта для оплаты.

**Подготовка приложения:**

Генерация Keystore для подписи приложения (release key). Сборка APK/AAB: npx cap build android --release. App Bundle (AAB) предпочтительнее APK для оптимизации размера.

**Требования к материалам:**

Название приложения: «Запаркуй — Аренда парковок» (до 30 символов). Краткое описание: до 80 символов. Полное описание: до 4 000 символов (ключевые слова). Иконки: 512×512 (магазин), дополнительные размеры для разных устройств. Скриншоты: минимум 2 скриншота (1080×1920 или 720×1280). Рекомендуется: 8 скриншотов, показывающих основные функции. Политика конфиденциальности: ссылка на размещённый документ ( обязательно).

**Контент в приложении:**

Возрастной рейтинг: Everyone (приложение не содержит возрастных ограничений). Категория: Auto & Vehicles / Parking. Теги: parking, real estate, moscow, apartment, car.

**Процесс публикации:**

Регистрация в Play Console → Загрузка AAB → Заполнение метаданных → Отправка на review → Ожидание review (1–3 дня) → Релиз (manual или gradual rollout). Обычно приложения review за 1–3 дня, но иногда дольше.

### 10.2. Apple App Store

**Требования к аккаунту:**

Apple Developer Program — 99 USD/год. Active Apple ID. Банковская карта, поддерживающая международные платежи.

**Подготовка приложения:**

Генерация Provisioning Profiles и сертификатов. Сборка iOS: npx cap build ios. Подготовка архива для загрузки в Xcode.

**Требования к материалам:**

Название: «Запаркуй» (до 30 символов). Подзаголовок: Аренда парковок в ЖК (до 30 символов). Описание: до 4 000 символов. Keywords: parking, moscow, rent, auto (до 100 символов). Иконки: 1024×1024 (App Store), набор иконок всех требуемых размеров. Скриншоты: 2–5 для iPhone (6.7", 6.5", 5.5"), опционально для iPad. Preview-видео: до 30 секунд (опционально).

**Особенности:**

Безопасность данных: приложение использует第三方 API — требуется раскрыть какие данные собираются. App Tracking Transparency: при использовании трекинга — запросить разрешение.

**Процесс публикации:**

Создание записи в App Store Connect → Загрузка через Xcode или Transporter → Заполнение метаданных → Выбор release strategy → Отправка на review → Ожидание review (24–48 часов для простых приложений) → Релиз.

### 10.4. Common pitfalls

Обе платформы могут отклонить приложение по следующим причинам:

Неполные метаданные: недостаточное описание, отсутствующая политика конфиденциальности. Вводящее в заблуждение описание: обещания, которые не выполняются. Проблемы с функциональностью: crash при запуске, неработающие основные функции. Несоответствие гайдлайнам: нестандартная навигация, проблемы с accessibility.

**Как избежать:**

Перед подачей: полное-functional testing на реальных устройствах. Подготовить чек-лист соответствия. Иметь размещённую политику конфиденциальности. Использовать стандартные паттерны навигации.
